# 클로드코드 하네스 엔지니어링 가이드
## 스스로 학습하고 실수를 성장의 양분으로 만드는 시스템

---

## 1. 하네스란 무엇인가

하네스(Harness)는 말(馬)에 채우는 마구(馬具)에서 온 비유다. 말의 힘 자체를 바꾸는 게 아니라, 그 힘을 원하는 방향으로 쓸 수 있게 만드는 장치다.

클로드코드에서 하네스란: **모델(Claude)의 지능을 그대로 두고, 도구·맥락·규칙·권한을 설계해서 원하는 결과를 안정적으로 뽑아내는 시스템**이다.

> "하네스가 Claude를 똑똑하게 만드는 게 아니다. Claude는 이미 똑똑하다. 하네스는 Claude에게 손, 눈, 작업 공간을 준다."
> — shareAI-lab/learn-claude-code

**핵심 원칙: 지능은 모델이 담당하고, 당신은 세계(환경)를 설계한다.**

---

## 2. 하네스의 두 가지 레이어

좋은 하네스에는 반드시 두 개의 레이어가 있다:

### 레이어 1: 가이드 (Feedforward — 실행 전 제어)
- CLAUDE.md 규칙
- 설계 문서, 예시, 디자인 스펙
- Plan mode에서의 계획 승인
- 권한 제한 (어떤 도구를 쓸 수 있는지)

### 레이어 2: 센서 (Feedback — 실행 후 자기교정)
- 린터, 테스트, 드리프트 스캔
- Learnings.md / MISTAKES.md 기록
- 프로세스 회고 (retrospective)
- 독립 검증자 (같은 에이전트가 자기 코드를 검증하면 안 됨)

**레이어 2 없이 레이어 1만 있으면, 하네스는 시간이 지날수록 열화(劣化)된다.**

## 3. 자기학습 시스템 설계

### 3.1 핵심 구조: Learnings Loop

```
세션 시작 → Learnings.md 읽기 → 작업 수행 → 실수/발견 기록 → 세션 종료
    ↑                                                              ↓
    └──────── 다음 세션에서 기록된 내용 반영 ←──────────────────────┘
```

이것이 "자기학습 루프"의 전부다. 복잡한 인프라가 아니라, **파일 하나에 쓰고 읽는 순환**이다.

### 3.2 파일 구조

```
프로젝트 루트/
├── CLAUDE.md                    ← 핵심 규칙 (고정 이름, 고정 위치)
├── .claude/
│   ├── commands/
│   │   └── update-learnings.md  ← 슬래시 커맨드 (/update-learnings)
│   └── settings.json
├── docs/
│   ├── LEARNINGS.md             ← 세션 간 누적 학습 기록
│   ├── MISTAKES.md              ← 실수 전용 기록
│   ├── DECISIONS.md             ← 주요 결정 + 이유
│   └── LAST_SESSION.md          ← 마지막 세션 상태 (이어하기용)
```

### 3.3 각 파일의 역할

| 파일 | 용도 | 누가 쓰나 | 누가 읽나 |
|------|------|----------|----------|
| CLAUDE.md | 전체 규칙, 워크플로우, 메타규칙 | 사람 + Claude | Claude (매 세션 자동) |
| LEARNINGS.md | 작동한 것, 실패한 것, 코드베이스 특이사항 | Claude | Claude (매 세션 시작) |
| MISTAKES.md | 반복된 실수 패턴, 금지 목록 | Claude | Claude (매 세션 시작) |
| DECISIONS.md | 왜 이렇게 결정했는지 (맥락 보존) | Claude | Claude (필요시) |
| LAST_SESSION.md | 진행상황, 다음 할 일, 블로킹 이슈 | Claude | Claude (세션 시작시) |

## 4. CLAUDE.md 작성 원칙

### 4.1 적게 쓸수록 좋다

- 프론티어 모델이 안정적으로 따르는 지시는 **약 150~200개**
- Claude Code 시스템 프롬프트가 이미 ~50개의 지시를 차지
- 즉 CLAUDE.md에는 **100개 이하의 지시**만 넣어야 효과적
- 이상적으로는 **200줄 이하, 60줄 이내**가 최적

> "지시가 많아질수록, 모든 지시의 준수율이 균일하게 떨어진다. 새 지시만 무시되는 게 아니라, 기존 지시도 함께 무시된다."
> — HumanLayer

### 4.2 코드 스타일은 넣지 마라

린터가 할 일을 LLM에게 시키지 마라. CLAUDE.md에 코드 스타일 가이드를 넣으면:
- 컨텍스트 윈도우 낭비
- 지시 준수율 하락
- 속도 저하

대신 **Hook으로 린터/포매터를 자동 실행**시키고, 에러만 Claude에게 보여줘라.

### 4.3 메타규칙을 포함시켜라

CLAUDE.md에 "규칙을 어떻게 쓰는지"에 대한 규칙을 넣어라. 이렇게 하면 Claude가 새 규칙을 추가할 때 일관된 품질을 유지한다.

```markdown
## META — 이 문서 유지 규칙
### 새 규칙 작성 원칙
- 구체적이고 검증 가능하게 작성 (나쁜 예: "조심해라" / 좋은 예: "X 함수에서 Y 패턴 금지")
- 하나의 규칙은 하나의 행동만 지시
- Warning Signs 섹션은 반복 실수가 3회 이상일 때만 추가
- 새 규칙 추가 시 상단 요약 섹션도 함께 업데이트
```

### 4.4 Compaction에서 살아남는 건 CLAUDE.md뿐

`/compact` 실행 후 Claude는 CLAUDE.md를 디스크에서 다시 읽는다. 대화에서만 언급한 지시는 사라진다. **중요한 건 반드시 CLAUDE.md에 기록**해야 한다.

## 5. 자기학습 루프 구현

### 5.1 CLAUDE.md에 넣을 워크플로우 규칙

```markdown
## 세션 시작 규칙
1. docs/LAST_SESSION.md를 읽고 이전 진행상황 파악
2. docs/LEARNINGS.md를 읽고 3~5줄로 요약한 뒤 작업 시작
3. docs/MISTAKES.md를 읽고 동일 실수 반복하지 않기

## 작업 중 규칙
- 에러 발생 시 즉시 docs/MISTAKES.md에 기록 (세션 끝이 아니라 그 즉시)
- 새로운 발견/패턴은 즉시 docs/LEARNINGS.md에 기록
- 계획과 달라진 결정이 있으면 docs/DECISIONS.md에 이유와 함께 기록

## 세션 종료 규칙
1. docs/LAST_SESSION.md 업데이트 (완료된 것, 다음 할 일, 블로킹 이슈)
2. docs/LEARNINGS.md에 미기록 항목 확인 후 추가
3. 이 단계를 건너뛰지 마라. 아무 발견이 없어도 "특이사항 없음"으로 기록

## Verify 체크리스트 (모든 작업 완료 후)
1. 코드 정상 작동 확인
2. MISTAKES.md 업데이트 여부 확인
3. LEARNINGS.md 업데이트 여부 확인
4. 위 기록을 안 했으면 지금 해라
```

### 5.2 슬래시 커맨드 활용

`.claude/commands/update-learnings.md` 파일을 만들어라:

```markdown
이 세션의 작업을 검토하라.
새로운 학습 — 패턴, 실수, 발견, 기억할 가치가 있는 해결책을 식별하라.
docs/LEARNINGS.md를 적절한 섹션에 간결한 항목으로 업데이트하라.
각 항목에 오늘 날짜를 포함하라.
```

이제 `/update-learnings`를 입력하면 언제든 학습 기록을 갱신할 수 있다.

### 5.3 기록 품질 규칙

**좋은 기록:**
```markdown
## 2026-04-07 — Supabase RLS 정책 순서
auth.uid()를 먼저 체크하는 정책이 있으면 anon 접근이 차단됨.
→ anon용 정책을 별도로 만들어야 함.
```

**나쁜 기록:**
```markdown
## Supabase 주의
RLS 설정 시 조심할 것.
```

원칙: **모호한 기록은 컨텍스트만 낭비하고 행동을 바꾸지 못한다. 모든 항목은 구체적이고 실행 가능해야 한다.**

## 6. 가비지 컬렉션 (Garbage Collection)

학습 파일은 계속 커진다. 관리하지 않으면 컨텍스트 윈도우를 잡아먹고 오히려 성능이 떨어진다.

### 6.1 Self-Pruning 규칙

CLAUDE.md에 추가:

```markdown
## 가비지 컬렉션 규칙
- 세션 시작 시 LEARNINGS.md가 50줄을 초과하면 정리 작업 수행
- 정리 = 중복 병합, 잘못 분류된 항목 이동, 오래된 항목 압축
- 14일 이상 된 세션 로그는 월간 요약으로 압축
- 핵심 원칙: 정리는 밀도를 높일 뿐, 지식을 삭제하지 않는다
- 압축/이동/승격만 허용, 완전 삭제 금지
```

### 6.2 승격 시스템 (Promotion)

```
1회 관찰 → 세션 로그에만 기록
2회 반복 → LEARNINGS.md로 승격
3회 이상 반복 → CLAUDE.md 규칙으로 승격 검토
해결된 이슈 → RESOLVED 섹션으로 이동 (삭제 아님)
```

### 6.3 라우팅 규칙

학습이 발생하면 어디에 기록할지 판단:

```
프로젝트 전용? → docs/LEARNINGS.md
글로벌 패턴 (2회 이상 반복)? → ~/.claude/CLAUDE.md (글로벌 규칙)
일회성 관찰? → 세션 로그에만 기록 (반복되면 승격)
확신 없으면? → 프로젝트 레벨에 기록
```

## 7. Hook 활용

Hook은 특정 이벤트에서 자동으로 스크립트를 실행하는 장치다.

### 7.1 유용한 Hook 패턴

**세션 시작 Hook (SessionStart):**
- docs/ 폴더의 학습 파일들 자동 로드
- 파일 크기 체크 → 50줄 초과 시 정리 경고

**Compact 전 Hook (PreCompact):**
- 미기록 학습을 LEARNINGS.md에 저장
- LAST_SESSION.md 업데이트
- 현재 진행 상태 스냅샷

**커밋 후 Hook (PostCommit):**
- 변경사항 요약을 세션 로그에 자동 기록

### 7.2 검증은 조용하게, 실패만 시끄럽게

테스트/린트 결과를 전부 보여주면 컨텍스트 윈도우가 넘친다. **성공은 조용히, 실패만 출력하라.**

```markdown
## Hook 출력 규칙
- 테스트 통과: 출력 삼키고 "✓ 테스트 통과"만 표시
- 테스트 실패: 에러 메시지 전문 출력
- 빌드 성공: 무음
- 빌드 실패: 에러 로그 출력
```

## 8. 자동강제 시스템 (Enforcement)

규칙을 만들어도 Claude가 안 따르면 소용없다. 강제하는 방법들:

### 8.1 워크플로우에 내장

규칙을 별도로 나열하지 말고, **워크플로우 단계 자체에 기록 행위를 포함**시켜라:

```markdown
## 작업 워크플로우
1. Investigate — 관련 파일 읽기 + LEARNINGS.md 확인
2. Plan — 계획 작성 후 사용자 승인 대기
3. Execute — 코드 작성 + 에러 발생 시 즉시 MISTAKES.md 기록
4. Verify — 테스트 + 기록 완료 확인 + LAST_SESSION.md 갱신
```

### 8.2 필수 구문 사용

Claude는 "하면 좋다" 수준의 지시를 자주 건너뛴다. **무조건적 구문**을 써라:

```markdown
# 약한 구문 (잘 안 지켜짐)
- 가능하면 LEARNINGS.md를 업데이트해주세요

# 강한 구문 (잘 지켜짐)
- LEARNINGS.md 업데이트는 필수다. 새 발견이 없어도 "특이사항 없음"을 기록하라.
- 이 단계를 건너뛰면 작업 미완료로 간주한다.
```

### 8.3 요약 강제

세션 시작 시 Learnings 파일을 "읽어라"만 하면 건너뛸 수 있다. **요약하라**고 하면 실제로 내용을 처리한다:

```markdown
## 세션 시작
1. docs/LEARNINGS.md를 읽고 3~5개 핵심 포인트로 요약하라
2. 요약을 사용자에게 보여준 뒤 작업 시작
```

## 9. MISTAKES.md 작성 가이드

실수 기록은 하네스에서 가장 높은 가치를 가진다.

### 9.1 포맷

```markdown
## [날짜] — [실수 한 줄 요약]
**상황**: 어떤 작업을 하고 있었는지
**잘못한 것**: 구체적으로 무엇을 잘못했는지
**올바른 방법**: 앞으로 어떻게 해야 하는지
**반복 횟수**: 이 실수가 몇 번 발생했는지
```

### 9.2 예시

```markdown
## 2026-04-07 — React state에서 장바구니 관리 시도
**상황**: 결제 플로우 구현 중 장바구니 상태 관리
**잘못한 것**: 로컬 컴포넌트 state로 장바구니를 관리하려 함
**올바른 방법**: 장바구니 데이터는 3개 컴포넌트에서 공유됨. 반드시 Zustand store (cartStore.ts) 사용
**반복 횟수**: 3회 → CLAUDE.md 규칙 승격 대상
```

### 9.3 부정적 학습이 더 가치 있다

"이렇게 하면 된다"보다 "이렇게 하면 안 된다"가 더 효과적이다. 실패한 접근법, 막다른 길, 피해야 할 패턴을 적극적으로 기록하라.

## 10. 실전 운영 팁

### 10.1 주간 정리 루틴

매주 한 번:
1. LEARNINGS.md 리뷰 — 모순되는 항목 없는지 확인
2. 반복 3회 이상 패턴 → CLAUDE.md 규칙 승격
3. 해결된 이슈 → RESOLVED 섹션으로 이동
4. 50줄 초과 시 압축

### 10.2 사람이 반드시 해야 하는 것

Claude가 기록한 내용을 **반드시 사람이 검토**하라. Claude는:
- 실패 원인을 잘못 귀인할 수 있다
- 한 번의 사례에서 과도하게 일반화할 수 있다
- 기술적으로는 맞지만 실전에서 오해를 부르는 기록을 남길 수 있다

**파일은 현실을 반영해야 한다. Claude의 해석만 반영하면 안 된다.**

### 10.3 하네스 최적화 > 하네스 완벽

> "하네스 셋업 최적화에 코드 출하보다 더 많은 시간을 쓰는 건 완전히 가능하다 — 우리도 그랬다. 원칙: 출하에 편향하라."
> — HumanLayer

하네스는 도구이지 목적이 아니다. 처음에는 간단하게 시작하고, 실제로 반복되는 문제가 생길 때 규칙을 추가하라.

## 11. Kitchen Flow 프로젝트용 시작 템플릿

```markdown
# CLAUDE.md — Kitchen Flow

## 프로젝트 개요
Kitchen Flow는 비프랜차이즈 식당용 주방 교육 시뮬레이터 SaaS.
스택: React + TypeScript + Supabase + React Three Fiber

## 워크플로우 (모든 작업에 적용)
1. **Investigate** — 관련 파일 읽기 + docs/LEARNINGS.md + docs/MISTAKES.md 확인
2. **Plan** — 계획 작성 후 사용자 승인 대기 (plan mode)
3. **Execute** — 코드 작성. 에러 발생 시 즉시 docs/MISTAKES.md에 기록
4. **Verify** — 테스트 + 기록 확인 + docs/LAST_SESSION.md 갱신

## 필수 기록 규칙
```
