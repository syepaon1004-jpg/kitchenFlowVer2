# KitchenFlow — 패널 시스템 원칙서

> **작성일**: 2026-04-01  
> **목적**: 패널 시스템 구현 시 절대 위반 불가한 원칙과 기술 제약을 정의한다.  
> **변경이 필요하다고 판단되면 반드시 sjb(최상위 지휘관)에게 먼저 확인한다.**

---

## 1. CSS 3D 기술 제약

### 1.1 라이브러리 금지

순수 CSS 3D만 사용한다. `perspective` + `rotateX` + `translateZ` 조합.
Three.js, Babylon.js, react-three-fiber 등 모든 3D 라이브러리를 사용하지 않는다.

### 1.2 transform 순서

서랍, 바구니 등 Z축 이동 + 회전이 동시에 필요한 경우:
```
translateZ → rotateX  (이동 먼저, 방향 나중)
```
이 순서를 뒤집으면 회전 후 이동이 되어 의도한 위치에 도달하지 않는다.

### 1.3 hit-test 방식

CSS 3D 회전된 요소는 브라우저의 기본 클릭 hit-test에 잡히지 않는다.
모든 3D 공간의 클릭 감지는 **scene 레벨에서 `getBoundingClientRect()` 좌표 비교**로 처리한다.
`pointer-events` CSS 속성에 의존하지 않는다.

### 1.4 애니메이션

접기/펼기 전환 애니메이션: **0.6초**. CSS transition으로 구현.

### 1.5 DOM 구조

패널들은 HTML div **부모-자식 중첩** 구조. 자식 패널이 부모 패널 하단에서 회전.
편집과 미리보기가 **동일한 DOM 구조**를 사용하며, `rotateX` 값만 전환한다.

## 2. 패널 구조 원칙

### 2.1 패널 수 고정

패널은 **3장 고정**이다. 어드민이 추가/삭제할 수 없다.

| 패널 | 접힌 방향 | 역할 |
|------|----------|------|
| 패널 1 | 기준면 (기울어짐) | 상부 벽면 |
| 패널 2 | 앞으로 90° (수평) | 작업면 |
| 패널 3 | 아래로 90° (수직) | 하부 전면 |

### 2.2 패널 가시성

| 모드 | 주방 패널 3장 | 배경 이미지 | 장비 | 폴드냉장고 내부 패널 2장 |
|------|-------------|------------|------|----------------------|
| 편집 | **보임** | 보임 | 보임 | 보임 |
| 미리보기 | **존재하지만 안 보임** | 보임 | 보임 | **보임** (흰색) |
| 인게임 | **존재하지만 안 보임** | 보임 | 보임 | **보임** (흰색) |

핵심: 주방 패널은 "제거"가 아니라 "투명"이다. DOM에 존재하고 장비의 3D 위치를 결정하지만, 시각적으로 렌더링되지 않는다.
폴드 냉장고 내부 패널은 예외적으로 모든 모드에서 보인다 (흰색 판).

### 2.3 배치 제약

화구는 **패널 2에만** 배치 가능하다. 이것이 유일한 하드 제약이다.
다른 장비는 패널 1·2·3 어디에든 배치 가능하다.

## 3. 장비 원칙

### 3.1 장비는 2D 면이다

장비는 독립된 3D 오브젝트가 아니다. 패널 표면에 붙은 2D 면이다.
패널이 접히면 장비도 같이 접혀서 자연스럽게 3D 위치를 갖는다.

### 3.2 UI상에서 웍은 없다

화구 자체가 장비이다. 화구 위에 별도의 웍 그래픽이 올라가지 않는다.
화구 UI에 볶기/불조절 버튼이 직접 존재하고, 불 단계에 따라 화구 전체 색상이 변한다.

### 3.3 화구 불 단계

- 0: 꺼짐 (기본)
- 1: 약불
- 2: 강불
- 클릭마다 +1, 2에서 다시 0으로 순환

### 3.4 홀로그램

- 위치: 패널 1 영역, 화구와 동일한 X좌표
- 크기: 화구와 동일
- 생성 조건: 편집 모드에서는 **존재하지 않음**. 미리보기/인게임에서 **자동 생성**
- 외형: 흰색 반투명 사각형
- 내용: 재료 목록 (텍스트), 볶기 시 불 아이콘 효과

### 3.5 서랍(냉장고) 구조

- 외형: 은색 철판 + 까만색 손잡이
- 3레이어: 컨테이너(preserve-3d) > 내부 판(rotateX -90deg 고정) + face
- 열림: `translateZ` 값만 변화 (rotateX는 고정값, 애니메이션하지 않음)
- 내부 편집: 별도 영역에서 그리드 편집 (합치기/나누기/행열 추가삭제, 셀마다 재료)

### 3.6 폴드 냉장고 구조

- 외형: 은색 철판 + 까만색 손잡이
- 내부 패널: **2장 고정**, 흰색, 모든 모드에서 보임
- 내부 패널 회전: 가운데 기준 `rotateX(90deg)` → 수직으로 세워짐
- 내부 재료/바구니: bottom 기준 `rotateX(-90deg)` → 회전 상쇄 → 패널에 수직으로 서 있음
- 내부에 바구니 배치 가능

### 3.7 바구니 핵심 규칙

- 편집은 패널 위가 아닌 **별도 2배 확대 편집 영역**에서 수행
- 어떤 패널에 있든 **패널 2와 수평** 보정 (getBasketCorrection)
- 셀 세우기: `rotateX(-90deg)`, `transformOrigin: bottom center`
- 펼치기: maxRow 기준, row 작을수록 Z축 상승량 큼

### 3.8 선반

UI 전용. 기능 없음.

## 4. 워크플로우 원칙

### 4.1 편집 → 미리보기 → 저장

이 순서는 고정이다. 미리보기를 거치지 않고 저장할 수 없다.

### 4.2 미리보기에서의 위치 기록

미리보기 모드에서 패널 덩어리의 상하 위치를 조정한 후, 이 위치가 DB에 기록된다.
인게임은 이 기록된 위치를 기반으로 렌더링한다.

### 4.3 어드민 페이지

기존 "히트박스 편집" 탭(주방 레이아웃)을 **삭제**하고 새로운 "주방 레이아웃" 탭을 **새로 생성**한다.

## 5. 기존 KitchenFlow 코딩 원칙 (계속 유효)

아래 원칙은 패널 시스템에서도 그대로 적용된다:

1. **물리엔진 클라이언트 전용** — Zustand에서만 계산. 세션 중 DB write 금지.
2. **action_history 판별** — 별도 status 컬럼 추가 금지.
3. **assigned_order_id 묶음** — group_id 추가 금지.
4. **재료 인스턴스는 드롭 성공 후에만 생성** — 드래그 시작 시 생성 금지.
5. **any 타입 금지** — unknown + 타입 가드.
6. **Zustand 셀렉터에 인라인 filter/map 금지**
7. **CSS 변수는 gameVariables.css에 정의** — 인라인 하드코딩 금지.
8. **파일 수정 전 반드시 현재 코드 읽기**
9. **표면적 패치보다 근본 원인 해결**
10. **어드민/게임 컴포넌트 공유 금지** — shared에만 공용 컴포넌트.

## 6. 폐기되는 원칙 (패널 시스템으로 인해)

아래 원칙은 기존 히트박스 시스템에 해당하며, 패널 시스템에서는 **적용되지 않는다**:

| 폐기 원칙 | 이유 |
|-----------|------|
| 비율 좌표(0~1) 히트박스 저장 | 히트박스 자체가 사라짐. 장비 배치는 새 방식 |
| navigate FK 참조 (UUID) | 섹션 네비게이션 방식 자체가 변경 예정 |
| SVG 통일 렌더링 | SVG 히트박스 시스템 폐기 |
| equipment는 히트박스 위 컴포넌트 배치 | 장비가 패널 위 2D 면으로 직접 배치 |
| 슬라이드 클램프 img.offsetWidth 기준 | 파노라마 슬라이드 폐기 |
| drag_image_url vs overlay_image_url 분리 | 히트박스 이미지 시스템 폐기 |

## 7. 1차 구현 경계선

### 만든다 (껍데기)

- 패널 시스템 전체 (편집 + 미리보기 + 저장)
- 모든 장비의 배치, 외형, 고유 인터랙션
- 모든 버튼과 UI 요소

### 만들되 연결하지 않는다

- 볶기 버튼 (UI만, 웍 로직 미연결)
- 불조절 버튼 (UI만, 불 로직 미연결)
- 씻기 버튼 (UI만, 세척 로직 미연결)
- 홀로그램 재료 텍스트 (영역만, 데이터 미연결)

### 만들지 않는다

- 바구니/서랍 그리드 편집 영역
- 재료 드래그 앤 드롭 연결
- 장비 올려놓기 기능
- 장비 기준 섹션 네비게이션
- **장비 컴포넌트에 @dnd-kit droppable을 미리 설정하지 않는다.** 게임 인터랙션 방식은 2차에서 결정된다. (DnD → 클릭/터치 선택 방식으로 전환 예정)

## 8. 예정된 변경 사항 (2차 이후, 참고만)

게임 인터랙션이 드래그 앤 드롭에서 **클릭/터치 선택 방식**으로 전환 예정이다. 오브젝트를 선택하면 HUD에 표시되고, 다른 대상을 클릭하면 액션이 발생하는 구조. 1차에서는 이를 고려하지 않되, 장비 컴포넌트를 DnD에 종속적으로 설계하지 않는다.
