# CLAUDE.md - Kitchen Simulator Ver2 작업 규칙

## 🚨 절대 규칙 (모든 작업 전 반드시 준수)

### 1. 작업 전 프로젝트 파악 필수
- 어떤 작업이든 시작하기 전에 관련 파일과 디렉토리 구조를 먼저 확인한다
- `find`, `ls`, `cat` 등으로 실제 파일 내용을 직접 읽는다
- **절대 파일 경로, 함수명, 변수명, import 경로를 추측하지 않는다**
- 기존 코드의 패턴(네이밍, import 방식, 상태관리 패턴)을 파악하고 동일하게 따른다
- 새 컴포넌트 생성 시 같은 폴더의 기존 컴포넌트를 먼저 열어 패턴을 확인한다

### 2. 모르면 멈춘다
- 확실하지 않은 정보는 "확인이 필요합니다"라고 말한다
- 에러 원인을 추측하지 않는다 — 로그, 에러 메시지, 실제 코드를 근거로 판단한다
- Supabase 테이블 구조, RLS 정책 등이 불확실하면 직접 확인하거나 사용자에게 묻는다
- 외부 라이브러리 API가 불확실하면 node_modules 내 타입 정의를 확인하거나 사용자에게 묻는다

### 3. 근본 원인 해결
- 에러 발생 시 증상이 아닌 원인을 찾는다
- 수정 전 "왜 이 에러가 발생하는지"를 먼저 설명한다
- 임시 우회는 사용자가 명시적으로 요청한 경우에만 사용한다

## 📋 작업 프로세스 (모든 작업에 자동 적용)

### Phase 1: 파악 (Investigate)
```
수행할 것:
- 관련 파일/폴더 구조 확인 (ls, cat으로 직접 읽기)
- 기존 코드 패턴 파악 (네이밍, import, 상태관리 방식)
- package.json 의존성 확인
- 현재 에러가 있다면 에러 로그 전문 확인
- Supabase 관련 작업이면 해당 테이블/함수 구조 확인

출력: "현재 상태 파악 결과: ..."
```

### Phase 2: 계획 (Plan)
```
수행할 것:
- 수정/생성할 파일 목록 (정확한 경로)
- 각 파일에서 변경할 내용 요약
- 영향받는 다른 파일 목록
- 예상 위험 요소

출력: "작업 계획: ..."
→ 사용자 확인 후 진행
```

### Phase 3: 실행 (Execute)
```
수행할 것:
- 계획대로 코드 수정/생성
- 한 파일씩 순차적으로 진행
- 각 변경의 이유를 설명
```

### Phase 4: 검증 (Verify)
```
수행할 것:
- npx tsc --noEmit (타입 체크)
- npm run build (빌드 테스트)
- 변경한 파일 목록과 변경 내용 요약 보고

출력: "검증 결과: ..."
```

```
kitchen-simulator-Ver2/
├── public/
├── src/
│   ├── assets/              # 정적 자원
│   ├── components/          # UI 컴포넌트 (도메인별 분리)
│   │   ├── admin/           # 관리자 기능
│   │   ├── equipment/       # 주방 장비 관련
│   │   ├── game/            # 게임 플레이 관련
│   │   ├── layout/          # 레이아웃 컴포넌트
│   │   └── ui/              # 공통 UI 컴포넌트
│   ├── hooks/               # 커스텀 React 훅
│   ├── lib/                 # 핵심 비즈니스 로직
│   │   ├── hitbox/          # 충돌 감지 로직
│   │   ├── physics/         # 물리 시뮬레이션
│   │   └── recipe/          # 레시피 처리 로직
│   ├── pages/               # 페이지 컴포넌트
│   ├── stores/              # 상태 관리
│   └── types/               # TypeScript 타입 정의
├── supabase/
│   └── migrations/          # DB 마이그레이션 파일
├── docs/                    # 문서
└── dist/                    # 빌드 출력 (수정 금지)
```

## ⚠️ 금지 사항

- 존재하지 않는 파일/함수/변수를 참조하지 않는다
- `// @ts-ignore`, `as any`, `eslint-disable`를 임의로 사용하지 않는다
- 기존 코드 스타일과 다른 패턴을 도입하지 않는다
- 사용자가 요청하지 않은 리팩토링을 하지 않는다
- 에러를 숨기는 방식(빈 catch, 조건부 무시)으로 해결하지 않는다
- 확인 없이 패키지를 설치하거나 삭제하지 않는다
- dist/ 폴더를 직접 수정하지 않는다

## 📌 도메인별 작업 가이드

### 컴포넌트 작업 시
- 새 컴포넌트는 해당 도메인 폴더에 생성 (game 관련 → components/game/)
- 같은 폴더의 기존 컴포넌트 패턴(props 구조, 스타일링 방식)을 먼저 확인 후 따른다
- 공통 UI는 components/ui/에, 레이아웃은 components/layout/에 배치

### lib/ 핵심 로직 작업 시
- hitbox, physics, recipe 로직은 게임의 핵심이므로 변경 전 기존 동작을 정확히 이해한다
- 관련 types/ 정의를 먼저 확인한다
- 변경 시 영향받는 components/game/ 파일을 반드시 확인한다

### Supabase 작업 시
- 테이블 구조 변경은 supabase/migrations/에 마이그레이션 파일로 관리한다
- RLS 정책, 함수, 트리거 변경 시 기존 정책을 먼저 확인한다
- 로컬과 프로덕션 DB 차이에 주의한다

### 배포 (Netlify) 작업 시
- 빌드 명령: npm run build
- 배포 전 로컬에서 빌드 성공 확인 필수
- 환경변수 변경 시 Netlify 대시보드 설정도 안내한다

## 💬 소통 규칙

- 작업 시작 전 Phase 1-2의 결과를 먼저 보여주고 진행 여부를 확인받는다
- "~인 것 같습니다" 대신 "~입니다 (근거: [파일명:라인])" 형태로 말한다
- 모르는 것은 "확인이 필요합니다: [구체적 질문]"으로 명시한다
- 변경 후 반드시 Phase 4 검증 결과를 보고한다
- 여러 해결책이 있을 때 각각의 장단점을 제시하고 사용자가 선택하게 한다
